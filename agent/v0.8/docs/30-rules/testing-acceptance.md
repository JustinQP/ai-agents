# 测试与验收规则

## 1. 适用范围

功能开发、Bug 修复、接口、数据、配置、页面、AI 能力和部署相关修改都应按本文件确定验证方式。

目标是提供真实、可复查的完成证据，而不是只声明“已完成”。

## 2. 验收标准

开发前尽量明确可检查的验收标准。

不推荐：

```text
功能正常
体验良好
代码没有问题
```

推荐：

```text
合法参数可以成功创建任务
缺少必填参数时返回明确错误
重复请求不会生成重复数据
任务失败时记录失败原因
现有回归测试仍然通过
```

小任务也必须说明如何判断完成。

## 3. 选择验证方式

按修改内容选择最小但充分的验证组合：

- 静态检查：格式、Lint、类型、配置语法
- 单元测试：函数、规则、转换、解析和错误分支
- 集成测试：数据库、文件、缓存、队列和外部服务
- 接口测试：正常、非法、未认证、无权限和状态冲突
- 构建或启动检查：依赖、编译、打包和运行配置
- 回归测试：受影响的既有功能
- 人工验收：页面、交互、生成内容和复杂业务流程

优先使用项目已经提供的命令，不编造不存在的脚本。

## 4. 单元测试

核心逻辑的测试应根据风险覆盖：

- 正常输入
- 空值和缺失输入
- 非法输入
- 边界值
- 状态变化
- 错误分支
- 幂等或重复执行

修复 Bug 时，条件允许应增加能复现该问题的回归用例。

## 5. 接口与集成测试

新增或修改接口时检查：

- 正常请求与返回结构
- 参数缺失和非法参数
- 未登录、无权限和越权
- 资源不存在和状态不允许
- 数据写入、回滚和重复请求
- 外部服务超时或失败
- 兼容性和调用方影响

集成环境不可用时，说明缺失条件，并提供替代验证或人工步骤。

## 6. 数据与迁移验收

涉及数据模型或迁移时验证：

- 新环境能完成升级
- 已有数据能正确迁移
- 默认值和空值符合预期
- 索引和约束不会破坏合法数据
- 失败时不会留下半完成状态
- 回滚方式真实可行

不可逆迁移必须在执行前得到明确授权。

## 7. 前端验收

页面或交互修改至少检查：

- 加载、空数据和错误状态
- 表单校验
- 提交中和重复点击
- 接口失败提示
- 权限不足和资源不存在
- 长任务进度或等待反馈
- 目标浏览器、终端或屏幕尺寸

视觉或交互结果需要人工验收时，记录具体步骤和实际结果。

## 8. AI 功能验收

AI 功能除正常场景外，还应验证：

- 输入过长或包含敏感信息
- 模型返回空内容
- 非法 JSON 或格式不稳定
- 超时、限流和服务不可用
- 模型切换后的兼容性
- 检索结果为空或不相关
- 提示词注入内容
- 生成结果校验和人工确认边界
- 成本、重试和降级控制

不得只依赖单次“看起来不错”的输出判断功能通过。

## 9. 测试结果记录

执行验证后必须记录：

- 完整命令或验证方式
- 通过、失败或未执行
- 失败原因
- 警告及其影响
- 未覆盖的关键路径

示例：

```text
自动化验证：
- 命令：pytest -q
- 结果：通过，42 passed
- 警告：无

未执行：
- 集成测试未执行，当前环境缺少数据库服务
```

实际数字和结果必须来自真实输出。

## 10. 失败与警告处理

测试失败时先区分：

- 本次修改引入的问题
- 既有历史问题
- 环境或配置缺失
- 外部服务不可用
- 测试用例本身过期

本次修改引入的问题应优先修复。其他问题必须明确说明，不能伪装为通过。

对弃用 API、依赖冲突、安全提示、类型错误、配置无效和迁移警告不得直接忽略。

## 11. 人工验收

人工验收必须包含：

1. 前置条件
2. 操作步骤
3. 预期结果
4. 实际结果
5. 是否通过

不能只写“人工验证通过”。

## 12. 验收输出模板

```text
验证结果：

1. 自动化测试
- 执行命令：
- 结果：
- 失败或警告：

2. 人工验证
- 场景：
- 步骤：
- 实际结果：

3. 回归影响
- 已确认：
- 未确认：

4. 风险与后续
- 风险：
- 后续建议：
```

不适用的项目写明“不涉及”。未执行的验证不得写成已通过。
