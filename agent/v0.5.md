# AGENTS.md

## 0. 核心原则

本项目由 AI 辅助开发，但所有修改必须遵循：小步推进、目标明确、可回滚、可验证。

AI 不应一次性大范围重构，不应在未理解现有结构前新增复杂设计。

任何改动都必须服务于当前任务目标，不允许无边界扩展。

工程实现必须具备基本可维护性、可配置性、可观测性和可验证性，不允许只堆功能而忽略工程底线。

## 1. 任务执行流程

执行任务时，必须按以下顺序推进：

1. 理解任务目标
2. 识别影响范围
3. 给出简要执行计划
4. 修改相关文件
5. 进行必要验证
6. 总结修改结果

如果任务较大，应先拆分为多个小步骤，不要一次性完成所有内容。

如果任务存在不确定性，应先说明假设，再基于假设推进。

## 2. 阶段化开发流程

涉及新功能、复杂改造、版本迭代时，必须按阶段推进：

1. 需求分析
2. 功能清单
3. 开发计划
4. 任务拆分
5. 分步实现
6. 验证验收
7. 总结与提交准备

不得跳过需求分析直接写代码。

不得在没有功能清单和开发计划的情况下，直接进行大范围开发。

如果用户只要求实现一个明确的小问题，可以直接进入实现阶段，但仍需说明影响范围和验证方式。

## 3. 任务粒度规则

开发任务必须保持小粒度。

一个任务原则上只解决一个明确问题。

如果任务包含多个目标，应先拆分为多个子任务，并按顺序执行。

禁止在一个任务中同时完成：

* 功能开发
* 架构重构
* UI 调整
* 文档重写
* 测试体系改造
* 依赖升级
* 部署改造
* 安全改造

除非用户明确要求，否则不要把多个类型的工作混在一起。

## 4. 修改范围

只修改与当前任务直接相关的文件。

禁止因为“顺手优化”而修改无关代码、目录结构、配置文件或文档。

如发现明显问题，但不属于当前任务范围，应在总结中记录为“后续建议”，不要直接修改。

## 5. 文件与目录规则

新增文件前，必须先判断是否可以复用或修改已有文件。

禁止随意新增临时文件、重复文件、实验文件。

新增文件必须满足以下条件：

* 有明确用途
* 放在合适目录
* 命名清晰稳定
* 与现有项目结构一致

如果需要新增文档，应优先放入已有 docs 结构中，不要在 docs 下随意平铺文件。

## 6. 文档治理规则

文档必须服务于项目开发、维护、交付或复盘，不应为了记录过程而无限新增。

新增或修改文档前，必须先判断文档类型：

* 项目说明：如 README、快速开始、部署说明
* 架构设计：如系统架构、模块设计、接口设计
* 开发计划：如版本计划、任务拆分、迭代记录
* 规则规范：如编码规范、目录规范、AI 协作规则
* 问题复盘：如故障分析、踩坑记录、优化建议
* 交付材料：如验收文档、汇报材料、方案文档

如果已有同类文档，应优先更新已有文档，而不是新建新文档。

禁止出现多个内容高度相似、命名接近、职责重复的文档。

## 7. docs 目录规则

如果项目存在 docs 目录，应保持目录结构稳定。

推荐优先使用以下结构：

```text
docs/
  00-overview/        # 项目概览、背景、目标
  10-design/          # 架构设计、模块设计、接口设计
  20-plan/            # 版本计划、开发计划、任务拆分
  30-rules/           # 工程规范、协作规则、AI 规则
  40-review/          # 项目复盘、问题记录、优化建议
  90-archive/         # 归档内容、历史方案、废弃文档
```

新增文档时，应优先放入上述目录。

如果当前项目已有自己的 docs 结构，应优先沿用现有结构，不要强行替换。

禁止直接在 docs 根目录持续堆放大量零散文档。

## 8. 文档命名规则

文档命名应清晰、稳定、可长期维护。

推荐命名方式：

```text
主题-用途.md
版本号-主题.md
日期-主题.md
```

示例：

```text
v0.3-开发计划.md
目录生成-技术方案.md
2026-06-项目复盘.md
```

禁止使用含义模糊的文件名，例如：

```text
新建文档.md
方案.md
优化.md
记录.md
临时.md
最终版.md
最终最终版.md
```

如果文档是临时过程材料，应放入明确的计划目录或归档目录，不要散落在项目根目录。

## 9. 文档更新规则

修改文档时，应优先保持文档的主线清晰。

如果只是补充已有内容，应在原文档中更新。

如果是替代旧方案，应明确标记：

* 当前推荐方案
* 历史方案
* 废弃原因
* 迁移说明

禁止在不说明关系的情况下，同时保留多份互相冲突的方案文档。

## 10. 工程化能力底线规则

项目开发不得只完成表面功能，还必须满足基本工程化要求。

涉及后端、前端、脚本、服务、部署、数据处理等代码时，必须优先检查以下工程底线。

### 10.1 配置管理

禁止将环境相关配置硬编码在业务代码中。

以下内容必须通过配置文件、环境变量或统一配置模块管理：

* 服务端口
* 数据库连接
* Redis、消息队列、对象存储等外部服务地址
* 第三方 API 地址和密钥
* 模型服务地址、模型名称、超时时间
* 文件存储路径
* 日志级别
* 运行环境标识，如 dev、test、prod

禁止在代码中写死生产环境地址、账号、密码、Token、密钥。

如果新增配置项，必须说明默认值、用途和影响范围。

### 10.2 日志管理

关键流程必须有必要日志，日志应服务于问题定位，而不是堆砌无意义输出。

至少应在以下位置考虑日志：

* 服务启动和关闭
* 外部服务调用
* 任务开始、完成、失败
* 异常捕获
* 关键状态变更
* 重要用户操作
* 数据导入、导出、生成、删除

日志内容应包含必要上下文，例如：

* request_id、trace_id 或 task_id
* 用户或租户标识
* 模块名
* 操作类型
* 错误原因
* 耗时信息

禁止在日志中输出密码、Token、密钥、身份证、手机号等敏感信息。

如果项目已有日志规范，应优先沿用，不要自行新增另一套日志体系。

### 10.3 错误处理

禁止吞掉异常。

禁止所有错误统一返回模糊错误，例如：

```text
系统异常
操作失败
错误码 500
错误码 530
```

错误处理应尽量区分：

* 参数错误
* 权限错误
* 认证失败
* 资源不存在
* 状态冲突
* 外部服务失败
* 超时
* 数据库错误
* 文件处理错误
* 模型调用失败
* 系统内部异常

对用户展示的错误信息应清晰、可理解。

对开发者排查的问题，应在日志中保留足够上下文。

新增接口或功能时，应考虑失败场景，不只实现成功路径。

### 10.4 接口返回

接口返回结构应保持稳定。

同一项目内不应出现多套互相冲突的返回格式。

接口返回至少应明确：

* 是否成功
* 业务状态码
* 提示信息
* 返回数据
* 错误详情或追踪标识

禁止随意改变已有接口字段含义。

如确实需要调整接口结构，必须说明兼容性影响。

### 10.5 数据库与数据模型

数据库设计必须保持字段含义清晰、类型一致、约束明确。

新增或修改表结构时，必须考虑：

* 主键设计
* 唯一约束
* 索引
* 字段类型
* 默认值
* 是否允许为空
* 创建时间和更新时间
* 软删除或删除策略
* 数据迁移影响

禁止使用手机号、名称、标题等易变化字段作为主键。

同一含义字段在不同表中应尽量保持相同类型和命名。

涉及状态字段时，必须明确状态枚举和流转规则。

### 10.6 事务与一致性

涉及多表写入、多步骤状态变更、扣减库存、任务状态流转、文件与数据库关联时，必须考虑事务或补偿机制。

禁止只完成第一步成功路径，而不处理后续失败场景。

如操作无法强事务保证，应说明：

* 哪些步骤可能失败
* 失败后如何补偿
* 是否需要重试
* 是否会产生脏数据或孤立文件

### 10.7 外部服务调用

调用外部服务时，必须考虑：

* 超时时间
* 重试策略
* 失败降级
* 错误日志
* 返回值校验
* 网络异常
* 鉴权失败
* 服务不可用

禁止无超时调用外部服务。

禁止默认假设第三方服务一定成功。

如果调用大模型、OCR、对象存储、向量库、消息队列等服务，应明确失败后的处理方式。

### 10.8 测试与验证

新增核心逻辑时，应优先补充或更新测试。

至少应考虑：

* 正常路径
* 参数错误
* 空数据
* 权限或状态不满足
* 外部服务失败
* 边界条件

如果暂时不补测试，必须说明原因和人工验证方式。

禁止伪造测试结果。

禁止只说“已测试”，但不说明执行了什么命令或验证了什么场景。

### 10.9 依赖管理

禁止随意引入新依赖。

新增依赖前必须确认：

* 是否已有依赖可复用
* 是否真的必要
* 是否维护活跃
* 是否会明显增加项目复杂度
* 是否影响部署体积或启动速度
* 是否存在明显安全风险

新增依赖后，应同步更新依赖声明文件。

禁止为了一个小功能引入大型框架。

### 10.10 安全底线

涉及认证、权限、文件、用户输入、外部请求时，必须考虑安全风险。

至少应避免：

* 明文保存密码
* 日志输出敏感信息
* 接口缺少权限校验
* 文件上传不校验类型和大小
* 路径穿越
* SQL 注入
* 命令注入
* 任意文件读取
* 越权访问
* 将密钥提交到代码仓库

如新增管理接口、导入导出接口、文件处理接口，应特别检查权限和输入校验。

### 10.11 部署与运行

新增功能不应只在本地可用，还应考虑部署和运行环境。

涉及启动、配置、数据库、外部服务、文件路径、定时任务、队列任务时，必须考虑：

* 开发环境
* 测试环境
* 生产环境
* 私有化部署
* 离线部署
* 环境变量差异
* 启动顺序
* 服务健康检查

禁止依赖手工修改代码来切换环境。

禁止把本地路径、个人电脑路径写入项目代码。

### 10.12 可观测性

核心链路应具备基本可观测性，方便定位问题。

涉及任务型、生成型、异步型、调用链较长的功能时，应考虑：

* request_id
* task_id
* trace_id
* 调用耗时
* 成功失败状态
* 失败原因
* 重试次数
* 上下游服务返回摘要

如果当前项目还没有完整链路追踪系统，也应通过日志保留最基本的排查线索。

### 10.13 异步任务与后台任务

涉及异步任务、队列任务、定时任务时，必须考虑：

* 任务状态
* 失败重试
* 超时处理
* 幂等性
* 重复执行
* 任务取消
* 任务进度
* 异常日志

禁止只把任务丢到后台，而没有状态记录和失败处理。

### 10.14 文件处理

涉及上传、下载、解析、导出、临时文件生成时，必须考虑：

* 文件大小限制
* 文件类型校验
* 文件名安全
* 存储路径
* 临时文件清理
* 失败后的残留文件
* 下载权限
* 文件与数据库记录的一致性

禁止使用不可信文件名直接拼接路径。

禁止临时文件长期残留。

### 10.15 前端工程底线

涉及前端页面或交互时，必须考虑：

* 加载状态
* 空数据状态
* 错误状态
* 按钮重复点击
* 表单校验
* 接口异常提示
* 长任务进度反馈
* 用户误操作确认

禁止只实现正常数据显示，而不处理加载、失败、空数据等状态。

### 10.16 AI 功能底线

涉及大模型、智能生成、Agent、提示词、向量检索、知识库调用时，必须考虑：

* 输入长度控制
* 输出格式约束
* 超时处理
* 失败重试
* 成本控制
* 模型切换
* 提示词版本管理
* 结果校验
* 敏感信息保护
* 用户可理解的失败提示

禁止默认假设模型一定按要求返回。

禁止无校验地直接使用模型输出写入数据库或进入关键流程。

## 11. 代码规则

代码应保持简单、清晰、可维护。

优先沿用项目已有风格和结构，不引入不必要的新框架、新依赖或复杂抽象。

新增代码必须考虑：

* 命名清晰
* 职责单一
* 错误处理合理
* 不破坏现有功能
* 满足基本工程化底线

如果存在多种实现方式，优先选择最小可用、最容易验证的方案。

## 12. 验证规则

完成修改后，必须说明：

* 修改了什么
* 为什么这样改
* 如何验证
* 是否存在风险或未完成项

如项目已有测试、检查或启动命令，应优先使用项目现有方式进行验证。

如果无法运行验证命令，必须说明原因，不能伪造验证结果。

涉及工程化能力的修改，还应说明：

* 配置是否可切换
* 日志是否足够排查
* 错误是否可理解
* 是否影响数据库结构
* 是否影响部署
* 是否需要补充测试

## 13. 提交前自检

在准备提交代码前，必须进行自检：

* 是否只修改了任务相关文件
* 是否引入了不必要的新依赖
* 是否存在临时文件、调试代码、无用注释
* 是否破坏已有功能
* 是否完成必要验证
* 是否需要补充文档或测试
* 是否新增了不必要的零散文档
* 是否存在重复、冲突或过期文档
* 是否存在硬编码配置
* 是否存在敏感信息泄露
* 是否缺少关键错误处理
* 是否缺少必要日志
* 是否影响部署或环境切换

如果发现问题，应先修正，再总结结果。

## 14. 输出要求

任务完成后，必须输出简要总结，包括：

* 本次完成内容
* 修改文件列表
* 验证结果
* 风险或未完成项
* 后续建议

如果涉及文档修改，还必须说明：

* 新增或修改了哪些文档
* 文档放置位置是否合理
* 是否存在需要归档或合并的旧文档

如果涉及工程化能力，还必须说明：

* 配置、日志、错误处理、测试、部署等是否受到影响
* 哪些工程化能力已补充
* 哪些工程化能力暂未覆盖，原因是什么

总结应简洁准确，不要夸大完成范围。

## 15. 禁止事项

禁止以下行为：

* 未经说明的大规模重构
* 删除已有功能或配置
* 引入无关依赖
* 修改与任务无关的文件
* 随意新增文档或临时文件
* 伪造测试结果
* 为了通过检查而隐藏问题
* 在没有明确要求时重写项目结构
* 跳过需求分析直接进行复杂开发
* 把多个不同类型任务混在一次修改中完成
* 在 docs 根目录堆放大量零散文档
* 创建多个职责重复的方案文档
* 使用“最终版”“临时版”等不可维护命名
* 硬编码生产配置、账号、密码、Token 或密钥
* 无超时调用外部服务
* 无权限校验暴露敏感接口
* 无校验直接使用大模型输出
* 只实现成功路径，不处理失败场景
