# Codex / AGENTS / Skills 三层系统架构

## 1. 总体设计目标

本仓库的目标不是堆叠规则，而是构建一个可演进的 AI 行为控制系统：

- 控制 AI 行为稳定性，而不是增加约束密度
- 将复杂性分层，而不是集中在单一 prompt
- 支持未来模型能力增强后的自动降维

---

## 2. 三层系统结构

```text
Layer 1: Codex Global（行为层）
Layer 2: AGENTS.md（决策与路由层）
Layer 3: Skills（领域执行层）
Layer 4: Validator（机器约束层）
```

---

## 3. 各层职责定义

### 3.1 Codex Global（行为层）

职责：定义“如何思考”，不定义“做什么”。

包含：

- 思考方式（先理解再行动）
- 不确定性处理（假设显式化）
- 输出结构偏好（结论优先）

不包含：

- 项目工程规则
- 技术栈
- 文件结构
- 测试/部署细节

本质：稳定的认知策略，而非执行规范。

---

### 3.2 AGENTS.md（决策层）

职责：定义“做什么 + 用哪个 Skill”。

包含：

- 任务风险分级
- 任务执行流程
- Skill 路由规则
- 冲突优先级

本质：调度器（router），而不是实现细节。

---

### 3.3 Skills（执行层）

职责：定义“如何做某一类事情”。

按领域拆分：

- testing
- database
- security
- ai-features
- deployment
- frontend
- external-services

本质：可组合的执行模块。

---

### 3.4 Validator（约束层）

职责：机器强校验，而非语言约束。

包含：

- 配置一致性检查
- 路径引用检查
- 规则冲突检测
- 行数/复杂度限制
- 结构 smell 检测

本质：替代“写在 prompt 里的规则”。

---

## 4. 信息流模型

```text
User Request
    ↓
Codex Global（行为方式）
    ↓
AGENTS.md（任务解析 + Skill 路由）
    ↓
Skills（具体执行逻辑）
    ↓
Validator（结构校验）
    ↓
Output
```

---

## 5. 关键设计原则

### 5.1 约束必须分层

- 行为约束（Global）
- 决策约束（AGENTS）
- 执行约束（Skills）
- 结构约束（Validator）

禁止在同一层重复定义。

---

### 5.2 规则优先“可删除性”

任何规则必须满足：

- 是否可以被 Skill 替代？
- 是否可以被 Validator 替代？
- 是否可以被删除而不影响行为？

不能通过以上测试的规则应降级或移除。

---

## 6. 演进方向（未来 v1.x）

### 6.1 从“规则系统”到“行为系统”

减少 prompt rule density，增强结构控制。

---

### 6.2 从“静态约束”到“动态校验”

更多规则迁移到 Validator。

---

### 6.3 从“人工维护规则”到“自检系统”

引入：

- 自动冲突检测
- Skill 覆盖率检测
- AGENTS 路由完整性检测

---

## 7. 核心结论

> 这个系统的核心不是“规则更多”，而是“规则分层 + 可验证”。
