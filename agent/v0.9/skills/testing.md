# 测试与验收

## 1. 加载条件

代码行为、Bug、接口、配置、数据、页面、AI 能力或部署结果发生变化时读取。

## 2. 验收标准

验收标准必须可检查。使用具体行为描述，不使用“功能正常”“体验良好”“代码没问题”等空泛结论。

示例：

```text
合法参数可以成功创建任务
缺少必填参数时返回明确错误
重复请求不会生成重复数据
任务失败时记录失败原因
受影响的回归测试仍然通过
```

## 3. 验证组合

按风险选择最小但充分的组合：

- 静态检查：格式、Lint、类型和配置语法
- 单元测试：规则、转换、解析、边界和错误分支
- 集成测试：数据库、文件、缓存、队列和外部服务
- 接口测试：正常、非法、未认证、无权限和状态冲突
- 构建或启动：依赖、编译、打包和运行配置
- 回归测试：受影响的既有功能
- 人工验收：页面、交互、生成内容和复杂流程

优先使用仓库已有命令，不编造脚本。

## 4. 重点场景

Bug 修复应尽量增加能复现问题的回归用例。

接口变更至少检查：

- 参数缺失和非法参数
- 未登录、无权限和越权
- 资源不存在和状态不允许
- 外部服务失败或超时
- 返回结构和调用方兼容性

数据变更至少检查升级、历史数据、失败状态和回滚。

## 5. 结果记录

```text
自动化验证：
- 命令：
- 结果：
- 失败或警告：

人工验证：
- 前置条件：
- 步骤：
- 预期结果：
- 实际结果：

未验证项：
- 
```

结果和数量必须来自真实输出。环境不可用时说明缺失条件和替代验证方式，不得包装成通过。
