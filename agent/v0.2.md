# AGENTS.md

## 0. 核心原则

本项目由 AI 辅助开发，但所有修改必须遵循：小步推进、目标明确、可回滚、可验证。

AI 不应一次性大范围重构，不应在未理解现有结构前新增复杂设计。

任何改动都必须服务于当前任务目标，不允许无边界扩展。

## 1. 任务执行流程

执行任务时，必须按以下顺序推进：

1. 理解任务目标
2. 识别影响范围
3. 给出简要执行计划
4. 修改相关文件
5. 进行必要验证
6. 总结修改结果

如果任务较大，应先拆分为多个小步骤，不要一次性完成所有内容。

如果任务存在不确定性，应先说明假设，再基于假设推进。

## 2. 修改范围

只修改与当前任务直接相关的文件。

禁止因为“顺手优化”而修改无关代码、目录结构、配置文件或文档。

如发现明显问题，但不属于当前任务范围，应在总结中记录为“后续建议”，不要直接修改。

## 3. 文件与目录规则

新增文件前，必须先判断是否可以复用或修改已有文件。

禁止随意新增临时文件、重复文件、实验文件。

新增文件必须满足以下条件：

* 有明确用途
* 放在合适目录
* 命名清晰稳定
* 与现有项目结构一致

如果需要新增文档，应优先放入已有 docs 结构中，不要在 docs 下随意平铺文件。

## 4. 代码规则

代码应保持简单、清晰、可维护。

优先沿用项目已有风格和结构，不引入不必要的新框架、新依赖或复杂抽象。

新增代码必须考虑：

* 命名清晰
* 职责单一
* 错误处理合理
* 不破坏现有功能

如果存在多种实现方式，优先选择最小可用、最容易验证的方案。

## 5. 文档规则

文档只在必要时新增或修改。

修改文档时，应优先更新已有文档，而不是新建类似主题的新文档。

新增文档前，应先确认：

* 是否已有相关文档
* 是否可以合并到已有文档
* 是否需要进入 docs 目录
* 文件名是否能长期维护

禁止为了记录过程而创建大量零散文档。

## 6. 验证规则

完成修改后，必须说明：

* 修改了什么
* 为什么这样改
* 如何验证
* 是否存在风险或未完成项

如项目已有测试、检查或启动命令，应优先使用项目现有方式进行验证。

如果无法运行验证命令，必须说明原因，不能伪造验证结果。

## 7. 输出要求

任务完成后，必须输出简要总结，包括：

* 本次完成内容
* 修改文件列表
* 验证结果
* 后续建议

总结应简洁准确，不要夸大完成范围。

## 8. 禁止事项

禁止以下行为：

* 未经说明的大规模重构
* 删除已有功能或配置
* 引入无关依赖
* 修改与任务无关的文件
* 随意新增文档或临时文件
* 伪造测试结果
* 为了通过检查而隐藏问题
* 在没有明确要求时重写项目结构
