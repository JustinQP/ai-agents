# AI 开发流程规则

## 1. 适用范围

在以下任务中读取本文件：

- 新功能开发
- 多文件修改
- 复杂 Bug 修复
- 业务流程变化
- 架构或公共接口调整
- 版本迭代

明确、低风险的单点修改不需要完整展开本流程。

## 2. 任务开始前

先确认：

- 要解决的问题是什么
- 当前行为和目标行为分别是什么
- 成功标准是什么
- 涉及哪些模块、文件和外部依赖
- 是否影响接口、数据、配置、部署或兼容性
- 是否存在不可逆操作

信息不足时，先检查现有仓库。仍无法确认的内容应明确标记为假设，不得自行补成事实。

## 3. 任务拆分

任务较大时，按可独立验证的结果拆分，而不是按文档、前端、后端等机械分割。

一个子任务应满足：

- 只有一个明确目标
- 影响范围可解释
- 结果可单独验证
- 失败后容易回滚

完成功能必需的测试、配置、迁移和文档属于该功能的完整变更，不视为无关扩展。

以下事项通常应独立处理：

- 无关重构
- 大版本依赖升级
- 架构迁移
- 测试体系重建
- 与当前目标无关的性能优化

## 4. 计划要求

标准任务的计划应简短并可执行，至少包含：

```text
目标：
- 

影响范围：
- 模块或文件：
- 接口、数据、配置、部署影响：

实施步骤：
1. 
2. 

验证方式：
- 

风险或假设：
- 
```

禁止使用“全面优化”“完善所有功能”“重构全部代码”等不可验收描述。

## 5. 实施规则

实施时遵循：

- 先理解现有结构，再修改
- 优先采用最小、清晰、兼容的实现
- 每一步尽量保持项目可运行或可验证
- 不覆盖与任务无关的用户修改
- 不为了代码更“优雅”引入大型抽象或新框架
- 修改公共行为时同步检查调用方和回归影响
- 涉及迁移时同时考虑升级、回滚和历史数据

## 6. 不确定性处理

存在多种合理方案时，应说明选择依据，例如：

- 对现有结构影响最小
- 兼容性最好
- 风险最低
- 最容易验证和回滚
- 与项目现有约定一致

示例：

```text
当前假设：
- 保持现有接口字段不变
- 不调整数据库表结构
- 只修改当前服务，不修改调用方

如仓库事实与假设冲突，应停止并重新评估。
```

## 7. 中断条件

出现以下情况时，停止扩大修改范围：

- 缺少关键文件或配置
- 项目现状与需求明显冲突
- 数据迁移规则不完整
- 高风险操作没有明确授权
- 测试持续失败且原因无法确认
- 外部依赖不可用且没有替代验证方式
- 发现可能泄露敏感信息

中断时说明：

- 已完成内容
- 当前阻塞原因
- 已修改文件
- 潜在风险
- 建议的后续动作

## 8. 提交前检查

提交或交付前确认：

- 修改只服务当前目标
- 没有临时文件、调试代码或无关格式化
- 没有引入不必要依赖
- 没有硬编码环境配置或敏感信息
- 必要测试和检查已经执行
- 测试失败、警告和未验证项已经如实说明
- 文档、配置和迁移与代码保持一致

## 9. 结果总结

总结应区分：

- 已实际完成的内容
- 已实际执行的验证
- 由于环境等原因未执行的验证
- 已知风险
- 后续建议

禁止夸大完成范围，禁止把计划写成结果。
