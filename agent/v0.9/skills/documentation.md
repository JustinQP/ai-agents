# 文档治理

## 1. 加载条件

新增、重写、合并、归档、删除文档或调整文档目录时读取。

## 2. 新增前检查

先确认：

- 是否真的需要形成项目文件
- 是否已有同主题文档
- 是否可以直接更新或合并
- 文档的读者、用途和维护场景
- 应放入哪个现有目录
- 文件名是否清晰、稳定

仅用于当前对话展示的方案不一定需要写入仓库。

## 3. 目录与命名

优先沿用项目已有结构，不强行替换。项目根目录只保留少量高频入口文件。

文件名应表达稳定主题和用途，例如：

```text
系统架构.md
接口设计.md
v1.2-开发计划.md
2026-06-故障复盘.md
```

避免使用“方案.md”“临时.md”“最新.md”“最终版.md”“副本.md”等模糊名称。

## 4. 更新、合并与归档

- 补充已有主题时优先更新原文档。
- 新方案替代旧方案时说明当前方案、替代关系和迁移方式。
- 不保留职责相同、内容冲突且关系不明的多份文档。
- 合并后确认链接、索引和引用同步更新。
- 有历史价值的内容优先归档；删除前确认没有有效引用。
- 文档中的命令、路径和示例必须来自项目事实。

## 5. AI 生成文档

- 不为记录执行过程创建大量文件。
- 不把草稿自动当成正式文档。
- 不创建与项目无关的目录体系。
- 不在未检查已有内容时创建重复主题。
- 不使用营销化语言替代事实。
- 不写入账号、密钥、个人资料或不必要的内部信息。

## 6. 结果说明

说明新增、修改、归档、删除、文件位置、替代关系和未确认内容。没有发生的项目写明“不涉及”。
