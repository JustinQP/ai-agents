# AGENTS.md

## 0. 核心原则

本项目由 AI 辅助开发，但所有修改必须遵循：小步推进、目标明确、可回滚、可验证。

AI 不应一次性大范围重构，不应在未理解现有结构前新增复杂设计。

任何改动都必须服务于当前任务目标，不允许无边界扩展。

## 1. 任务执行流程

执行任务时，必须按以下顺序推进：

1. 理解任务目标
2. 识别影响范围
3. 给出简要执行计划
4. 修改相关文件
5. 进行必要验证
6. 总结修改结果

如果任务较大，应先拆分为多个小步骤，不要一次性完成所有内容。

如果任务存在不确定性，应先说明假设，再基于假设推进。

## 2. 阶段化开发流程

涉及新功能、复杂改造、版本迭代时，必须按阶段推进：

1. 需求分析
2. 功能清单
3. 开发计划
4. 任务拆分
5. 分步实现
6. 验证验收
7. 总结与提交准备

不得跳过需求分析直接写代码。

不得在没有功能清单和开发计划的情况下，直接进行大范围开发。

如果用户只要求实现一个明确的小问题，可以直接进入实现阶段，但仍需说明影响范围和验证方式。

## 3. 任务粒度规则

开发任务必须保持小粒度。

一个任务原则上只解决一个明确问题。

如果任务包含多个目标，应先拆分为多个子任务，并按顺序执行。

禁止在一个任务中同时完成：

* 功能开发
* 架构重构
* UI 调整
* 文档重写
* 测试体系改造
* 依赖升级

除非用户明确要求，否则不要把多个类型的工作混在一起。

## 4. 修改范围

只修改与当前任务直接相关的文件。

禁止因为“顺手优化”而修改无关代码、目录结构、配置文件或文档。

如发现明显问题，但不属于当前任务范围，应在总结中记录为“后续建议”，不要直接修改。

## 5. 文件与目录规则

新增文件前，必须先判断是否可以复用或修改已有文件。

禁止随意新增临时文件、重复文件、实验文件。

新增文件必须满足以下条件：

* 有明确用途
* 放在合适目录
* 命名清晰稳定
* 与现有项目结构一致

如果需要新增文档，应优先放入已有 docs 结构中，不要在 docs 下随意平铺文件。

## 6. 文档治理规则

文档必须服务于项目开发、维护、交付或复盘，不应为了记录过程而无限新增。

新增或修改文档前，必须先判断文档类型：

* 项目说明：如 README、快速开始、部署说明
* 架构设计：如系统架构、模块设计、接口设计
* 开发计划：如版本计划、任务拆分、迭代记录
* 规则规范：如编码规范、目录规范、AI 协作规则
* 问题复盘：如故障分析、踩坑记录、优化建议
* 交付材料：如验收文档、汇报材料、方案文档

如果已有同类文档，应优先更新已有文档，而不是新建新文档。

禁止出现多个内容高度相似、命名接近、职责重复的文档。

## 7. docs 目录规则

如果项目存在 docs 目录，应保持目录结构稳定。

推荐优先使用以下结构：

```text
docs/
  00-overview/        # 项目概览、背景、目标
  10-design/          # 架构设计、模块设计、接口设计
  20-plan/            # 版本计划、开发计划、任务拆分
  30-rules/           # 工程规范、协作规则、AI 规则
  40-review/          # 项目复盘、问题记录、优化建议
  90-archive/         # 归档内容、历史方案、废弃文档
```

新增文档时，应优先放入上述目录。

如果当前项目已有自己的 docs 结构，应优先沿用现有结构，不要强行替换。

禁止直接在 docs 根目录持续堆放大量零散文档。

## 8. 文档命名规则

文档命名应清晰、稳定、可长期维护。

推荐命名方式：

```text
主题-用途.md
版本号-主题.md
日期-主题.md
```

示例：

```text
v0.3-开发计划.md
目录生成-技术方案.md
2026-06-项目复盘.md
```

禁止使用含义模糊的文件名，例如：

```text
新建文档.md
方案.md
优化.md
记录.md
临时.md
最终版.md
最终最终版.md
```

如果文档是临时过程材料，应放入明确的计划目录或归档目录，不要散落在项目根目录。

## 9. 文档更新规则

修改文档时，应优先保持文档的主线清晰。

如果只是补充已有内容，应在原文档中更新。

如果是替代旧方案，应明确标记：

* 当前推荐方案
* 历史方案
* 废弃原因
* 迁移说明

禁止在不说明关系的情况下，同时保留多份互相冲突的方案文档。

## 10. 代码规则

代码应保持简单、清晰、可维护。

优先沿用项目已有风格和结构，不引入不必要的新框架、新依赖或复杂抽象。

新增代码必须考虑：

* 命名清晰
* 职责单一
* 错误处理合理
* 不破坏现有功能

如果存在多种实现方式，优先选择最小可用、最容易验证的方案。

## 11. 验证规则

完成修改后，必须说明：

* 修改了什么
* 为什么这样改
* 如何验证
* 是否存在风险或未完成项

如项目已有测试、检查或启动命令，应优先使用项目现有方式进行验证。

如果无法运行验证命令，必须说明原因，不能伪造验证结果。

## 12. 提交前自检

在准备提交代码前，必须进行自检：

* 是否只修改了任务相关文件
* 是否引入了不必要的新依赖
* 是否存在临时文件、调试代码、无用注释
* 是否破坏已有功能
* 是否完成必要验证
* 是否需要补充文档或测试
* 是否新增了不必要的零散文档
* 是否存在重复、冲突或过期文档

如果发现问题，应先修正，再总结结果。

## 13. 输出要求

任务完成后，必须输出简要总结，包括：

* 本次完成内容
* 修改文件列表
* 验证结果
* 风险或未完成项
* 后续建议

如果涉及文档修改，还必须说明：

* 新增或修改了哪些文档
* 文档放置位置是否合理
* 是否存在需要归档或合并的旧文档

总结应简洁准确，不要夸大完成范围。

## 14. 禁止事项

禁止以下行为：

* 未经说明的大规模重构
* 删除已有功能或配置
* 引入无关依赖
* 修改与任务无关的文件
* 随意新增文档或临时文件
* 伪造测试结果
* 为了通过检查而隐藏问题
* 在没有明确要求时重写项目结构
* 跳过需求分析直接进行复杂开发
* 把多个不同类型任务混在一次修改中完成
* 在 docs 根目录堆放大量零散文档
* 创建多个职责重复的方案文档
* 使用“最终版”“临时版”等不可维护命名
