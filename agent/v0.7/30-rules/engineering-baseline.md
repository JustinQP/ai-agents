## 1. 定位

本文件定义项目工程化能力底线。

适用于以下任务：

* 后端开发
* 前端开发
* 脚本开发
* 服务开发
* 数据处理
* 部署改造
* 配置改造
* AI 功能开发
* 外部服务接入
* 文件处理
* 数据库改造

目标是避免项目只堆功能，而忽略可维护性、可配置性、可观测性、可验证性和安全性。

## 2. 基本原则

工程实现必须满足：

* 配置可切换
* 日志可排查
* 错误可理解
* 接口可兼容
* 数据可维护
* 外部服务可失败
* 部署可落地
* 安全有底线
* 测试可验证

禁止只实现成功路径，不处理失败场景。

## 3. 配置管理

禁止将环境相关配置硬编码在业务代码中。

以下内容必须通过配置文件、环境变量或统一配置模块管理：

* 服务端口
* 数据库连接
* Redis、消息队列、对象存储等外部服务地址
* 第三方 API 地址和密钥
* 模型服务地址
* 模型名称
* 超时时间
* 文件存储路径
* 日志级别
* 运行环境标识，如 dev、test、prod

禁止在代码中写死：

* 生产环境地址
* 账号
* 密码
* Token
* API Key
* Secret
* 本地个人路径

如果新增配置项，必须说明：

* 配置名
* 默认值
* 用途
* 影响范围
* 是否需要更新部署配置

## 4. 日志管理

关键流程必须有必要日志。

日志应服务于问题定位，而不是堆砌无意义输出。

至少应在以下位置考虑日志：

* 服务启动和关闭
* 外部服务调用
* 任务开始、完成、失败
* 异常捕获
* 关键状态变更
* 重要用户操作
* 数据导入、导出、生成、删除
* 后台任务状态变化

日志内容应尽量包含：

* request_id
* trace_id
* task_id
* 用户或租户标识
* 模块名
* 操作类型
* 错误原因
* 耗时信息
* 上下游服务摘要

禁止在日志中输出：

* 密码
* Token
* 密钥
* 身份证
* 手机号
* 银行卡
* 其他敏感信息

如果项目已有日志规范，应优先沿用，不要自行新增另一套日志体系。

## 5. 错误处理

禁止吞掉异常。

禁止所有错误统一返回模糊错误，例如：

```text
系统异常
操作失败
错误码 500
错误码 530
```

错误处理应尽量区分：

* 参数错误
* 权限错误
* 认证失败
* 资源不存在
* 状态冲突
* 外部服务失败
* 超时
* 数据库错误
* 文件处理错误
* 模型调用失败
* 系统内部异常

对用户展示的错误信息应清晰、可理解。

对开发者排查的问题，应在日志中保留足够上下文。

新增接口或功能时，应考虑失败场景，不只实现成功路径。

## 6. 接口返回

接口返回结构应保持稳定。

同一项目内不应出现多套互相冲突的返回格式。

接口返回至少应明确：

* 是否成功
* 业务状态码
* 提示信息
* 返回数据
* 错误详情或追踪标识

禁止随意改变已有接口字段含义。

如确实需要调整接口结构，必须说明兼容性影响。

## 7. 数据库与数据模型

数据库设计必须保持字段含义清晰、类型一致、约束明确。

新增或修改表结构时，必须考虑：

* 主键设计
* 唯一约束
* 索引
* 字段类型
* 默认值
* 是否允许为空
* 创建时间
* 更新时间
* 软删除或删除策略
* 数据迁移影响

禁止使用手机号、名称、标题等易变化字段作为主键。

同一含义字段在不同表中应尽量保持相同类型和命名。

涉及状态字段时，必须明确：

* 状态枚举
* 状态含义
* 状态流转规则
* 非法状态如何处理

## 8. 事务与一致性

涉及以下场景时，必须考虑事务或补偿机制：

* 多表写入
* 多步骤状态变更
* 扣减库存
* 任务状态流转
* 文件与数据库关联
* 外部服务调用后落库
* 支付或计费
* 批量导入

禁止只完成第一步成功路径，而不处理后续失败场景。

如操作无法强事务保证，应说明：

* 哪些步骤可能失败
* 失败后如何补偿
* 是否需要重试
* 是否会产生脏数据或孤立文件

## 9. 外部服务调用

调用外部服务时，必须考虑：

* 超时时间
* 重试策略
* 失败降级
* 错误日志
* 返回值校验
* 网络异常
* 鉴权失败
* 服务不可用
* 幂等性

禁止无超时调用外部服务。

禁止默认假设第三方服务一定成功。

如果调用以下服务，应明确失败后的处理方式：

* 大模型
* OCR
* 对象存储
* 向量库
* 消息队列
* 数据库
* 缓存
* 支付服务
* 第三方 API

## 10. 依赖管理

禁止随意引入新依赖。

新增依赖前必须确认：

* 是否已有依赖可复用
* 是否真的必要
* 是否维护活跃
* 是否会明显增加项目复杂度
* 是否影响部署体积
* 是否影响启动速度
* 是否存在明显安全风险

新增依赖后，应同步更新依赖声明文件。

禁止为了一个小功能引入大型框架。

## 11. 安全底线

涉及认证、权限、文件、用户输入、外部请求时，必须考虑安全风险。

至少应避免：

* 明文保存密码
* 日志输出敏感信息
* 接口缺少权限校验
* 文件上传不校验类型和大小
* 路径穿越
* SQL 注入
* 命令注入
* 任意文件读取
* 越权访问
* 将密钥提交到代码仓库

如新增以下接口，应特别检查权限和输入校验：

* 管理接口
* 导入接口
* 导出接口
* 文件上传接口
* 文件下载接口
* 批量操作接口
* 删除接口
* AI 生成接口

## 12. 部署与运行

新增功能不应只在本地可用，还应考虑部署和运行环境。

涉及启动、配置、数据库、外部服务、文件路径、定时任务、队列任务时，必须考虑：

* 开发环境
* 测试环境
* 生产环境
* 私有化部署
* 离线部署
* 环境变量差异
* 启动顺序
* 服务健康检查

禁止依赖手工修改代码来切换环境。

禁止把本地路径、个人电脑路径写入项目代码。

## 13. 可观测性

核心链路应具备基本可观测性。

涉及任务型、生成型、异步型、调用链较长的功能时，应考虑：

* request_id
* task_id
* trace_id
* 调用耗时
* 成功失败状态
* 失败原因
* 重试次数
* 上下游服务返回摘要

如果当前项目还没有完整链路追踪系统，也应通过日志保留最基本的排查线索。

## 14. 异步任务与后台任务

涉及异步任务、队列任务、定时任务时，必须考虑：

* 任务状态
* 失败重试
* 超时处理
* 幂等性
* 重复执行
* 任务取消
* 任务进度
* 异常日志

禁止只把任务丢到后台，而没有状态记录和失败处理。

## 15. 文件处理

涉及上传、下载、解析、导入、导出、临时文件生成时，必须考虑：

* 文件大小限制
* 文件类型校验
* 文件名安全
* 存储路径
* 临时文件清理
* 失败后的残留文件
* 下载权限
* 文件与数据库记录一致性

禁止使用不可信文件名直接拼接路径。

禁止临时文件长期残留。

## 16. 前端工程底线

涉及前端页面或交互时，必须考虑：

* 加载状态
* 空数据状态
* 错误状态
* 提交中状态
* 按钮重复点击
* 表单校验
* 接口异常提示
* 长任务进度反馈
* 用户误操作确认

禁止只实现正常数据显示，而不处理加载、失败、空数据等状态。

## 17. AI 功能底线

涉及大模型、智能生成、Agent、提示词、向量检索、知识库调用时，必须考虑：

* 输入长度控制
* 输出格式约束
* 超时处理
* 失败重试
* 成本控制
* 模型切换
* 提示词版本管理
* 结果校验
* 敏感信息保护
* 用户可理解的失败提示

禁止默认假设模型一定按要求返回。

禁止无校验地直接使用模型输出写入数据库或进入关键流程。

## 18. 工程化总结要求

如果任务涉及工程化能力，完成后必须说明：

```text
工程化影响：
- 配置：
- 日志：
- 错误处理：
- 数据库：
- 外部服务：
- 安全：
- 部署：
- 测试：
```

不涉及的项目应写明“不涉及”。

## 19. 禁止事项

禁止以下行为：

* 硬编码生产配置、账号、密码、Token 或密钥
* 无超时调用外部服务
* 吞掉异常不记录
* 所有错误统一返回模糊错误
* 无权限校验暴露敏感接口
* 无校验直接使用大模型输出
* 只实现成功路径，不处理失败场景
* 使用本地个人路径作为项目配置
* 为小功能引入大型框架
* 在日志中输出敏感信息
