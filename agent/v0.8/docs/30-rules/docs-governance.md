# 文档治理规则

## 1. 适用范围

在以下任务中读取本文件：

- 新增或重写文档
- 调整文档目录
- 合并、归档或删除文档
- 生成设计方案、开发计划、规则或复盘材料

目标是让文档长期可维护，而不是记录所有临时过程。

## 2. 新增文档前

先确认：

- 是否真的需要形成项目文件
- 是否已有同主题文档
- 是否能直接更新已有文档
- 文档的读者和用途是什么
- 应放在哪个现有目录
- 文件名是否稳定、清晰
- 后续由谁在什么场景维护

仅用于当前对话展示的方案，不一定需要写入仓库。

## 3. 目录原则

优先沿用项目已有文档结构，不强行替换。

项目没有明确结构时，可参考：

```text
docs/
  00-overview/   # 项目说明、快速开始
  10-design/     # 架构、接口、数据设计
  20-plan/       # 版本计划、任务拆分
  30-rules/      # 工程、测试和协作规则
  40-review/     # 复盘、问题分析
  90-archive/    # 历史和废弃内容
```

项目根目录只保留少量高频入口文件，例如 `README.md`、`AGENTS.md` 和 `CHANGELOG.md`。

## 4. 命名规则

文件名应表达稳定主题和用途。

推荐：

```text
系统架构.md
接口设计.md
v1.2-开发计划.md
2026-06-故障复盘.md
测试验收规则.md
```

避免：

```text
方案.md
记录.md
临时.md
最新.md
最终版.md
最终最终版.md
副本.md
```

版本或日期仅在确有历史区分价值时使用。

## 5. 更新与冲突

- 补充已有主题时优先更新原文档。
- 新方案替代旧方案时，明确当前方案、替代关系和迁移说明。
- 不同时保留多份职责相同且互相冲突的文档。
- 代码、配置或接口发生变化时，同步检查相关文档是否失效。
- 文档中的命令、路径和示例必须来自项目事实，不得编造。

## 6. 合并与归档

以下情况优先合并：

- 多个文件描述同一方案
- 同一版本存在重复计划
- 标题不同但正文高度重合
- 新旧方案并存但关系不清

归档文档应说明：

- 归档原因
- 当前替代文档
- 是否仍有参考价值

历史资料有价值时优先归档，不直接删除。删除前应确认没有有效引用。

## 7. AI 生成文档要求

AI 生成文档时：

- 不为记录执行过程创建大量文件
- 不把草稿自动当成正式文档
- 不凭空创建与项目无关的目录体系
- 不在未检查已有文档时创建重复主题
- 不使用模糊或营销化语言代替事实
- 示例中的命令、文件名和配置应能在项目中对应
- 不写入个人资料、账号、密钥或不必要的内部信息

## 8. 修改完成后的说明

文档任务完成后说明：

```text
文档变更：
- 新增：
- 修改：
- 归档：
- 删除：

文档位置：
- 

替代或引用关系：
- 

未确认内容：
- 
```

没有发生的变更写明“不涉及”。
